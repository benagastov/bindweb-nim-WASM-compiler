# Migration: vendored binji blobs → from-source builds

## What was wrong

The previous in-browser IDE shipped prebuilt compiler binaries and called the
setup "reproduction." It wasn't reproduction — it was copying. The
`reproduce.sh` cloned `benagastov/Nim-WASM-Compiler` and copied
`clang.wasm`, `lld.wasm`, `memfs.wasm`, and `sysroot.tar` out of it.

Those four files are **byte-for-byte identical to binji's `wasm-clang`**
prebuilts (the CppCon 2019 demo, last built Dec 2023, LLVM 8.0.1-era):

| file        | MD5 (this project's old static/clang) | MD5 (binji `wasm-clang`) |
|-------------|----------------------------------------|--------------------------|
| clang.wasm  | `fd63fc9e39f1c08200518e6b59da5d81`     | `fd63fc9e39f1c08200518e6b59da5d81` |
| lld.wasm    | `4264aa35d1bc5fd10ba91d61d0048033`     | `4264aa35d1bc5fd10ba91d61d0048033` |
| memfs.wasm  | `9a155eb19e9010d0eeb4dce7d45e7054`     | `9a155eb19e9010d0eeb4dce7d45e7054` |
| sysroot.tar | `60410a2bff1b8578c9a929a457cb5d6b`     | `60410a2bff1b8578c9a929a457cb5d6b` |

Consequence: **nobody could modify the C/C++ compiler.** Editing LLVM and
rebuilding was impossible because there was no build — just an opaque 31 MB
`clang.wasm` someone else produced.

## What changed

1. **`toolchain/` is now a real from-source build system.** clang.wasm/lld.wasm/
   sysroot.tar are produced from pinned LLVM *source*; memfs.wasm from the actual
   `memfs.c`; nim.wasm/nim-bundle.js from Nim *source*. Every upstream commit is
   pinned in `toolchain/versions.env`.

2. **The blobs became build outputs.** `static/clang/*.wasm`, `sysroot.tar`, and
   `static/nim/*.wasm` are `.gitignore`d. A checkout contains source + JS glue
   only; `make toolchain` (or CI) produces the binaries.

3. **`reproduce.sh` → `fetch-toolchain.sh`.** No third-party clone, no blob copy.
   It either builds from source (`build`, default) or downloads your own
   from-source CI artifacts (`ci`).

4. **memfs is de-blobbed to source.** `toolchain/memfs/src/memfs.c` +
   `stb_sprintf.h` are vendored from binji's LLVM fork *as source*, with the
   exact build recipe from his Makefile. (binji's project is open source; this
   preserves attribution while making it buildable.)

5. **CI builds it for real.** `.github/workflows/build-toolchain.yml` compiles
   the whole toolchain from source on GitHub-hosted runners and publishes
   artifacts — so the binaries are reproducible outputs of a transparent
   process, not mystery files in git.

6. **Framework bug fixed.** `nim-bindweb/config.nims` hardcoded
   `-resource-dir=.../clang/8.0.1` (only valid against binji's LLVM 8). It now
   auto-detects the clang resource dir from the sysroot, so the framework builds
   against any clang/sysroot (overridable via `CLANG_RESOURCE_DIR`).

## What stayed

- The **JS glue** (`clang.js`, `bindweb-browser-runtime.js`, the IDE
  `index.html`) is source and stays in git. `clang.js` keeps the `--export-table`
  linker patch; that patch is now also applied in the from-source link flags, so
  source-built and glue stay consistent.
- `nim-bindweb/toolchain/wasi-sysroot.tar` (a wasi-libc sysroot: headers +
  `libc.a` + crt1.o + builtins) is kept as build *data* so the native framework
  build works out of the box. It is not a compiler; it can be regenerated by the
  clang-wasm pipeline (`sysroot.tar`) or swapped for a wasi-sdk sysroot.

## If you maintain a fork

- Bump a compiler: edit the pin in `toolchain/versions.env`, run `make toolchain`.
- Patch the C/C++ compiler: edit LLVM at the pinned source, rebuild
  `toolchain/clang-wasm`. This is now possible — that was the goal.
- Don't re-commit the `.wasm` outputs; let CI publish them.
